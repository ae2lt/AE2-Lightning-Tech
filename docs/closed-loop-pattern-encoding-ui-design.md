# 闭环样板编码 UI 交接设计稿

## 1. 范围

本文只定义天枢样板终端中“闭环样板”页的 UI 与菜单数据契约，供后续实现者接入。当前后端已经负责闭环发现、嵌套展开、计量分析、排序和最终编码；本设计不要求修改上传流程、闭环 payload UUID 或 `netOutputs` 结构。

## 2. 所需控件

本文不规定控件的排列、坐标、分页方式或视觉样式，只要求闭环编码页提供以下专用控件：

- **27 个可标记样板槽**：选择闭环成员样板并设置每个成员的执行份数。
- **9 个可标记输出槽**：最多标记 1 个主产物和 8 个副产物。超出 9 个的其他产物不参与本次主/副产物计算。
- **若干只读输入槽**：显示算法计算出的普通外部输入需求，玩家不能手工标记或修改。
- **若干只读种子输入槽**：显示算法计算出的可复用种子需求，玩家不能手工标记或修改。
- **2 个数字输入控件**：分别设置任务借种倍率和种子存储倍率。

27 个样板槽本身就是手工编码的成员数量硬边界。自动发现与手工编辑共用同一组样板槽，不再维护另一套只读成员列表。

## 3. 样板标记槽语义

每个非空槽保存一项草稿成员：

```text
SourcePatternSnapshot pattern
long copiesPerCycle
```

- 槽内物品必须是 AE2 能识别的已编码样板。
- 允许普通样板、模糊/过载样板以及作为宏引用的闭环样板。
- 允许完整闭环样板作为宏并在编码时展平；不接受执行层内部生成的成员快照。
- 槽上显示的数量就是 `copiesPerCycle`，必须大于等于 1，并允许超过物品原版堆叠上限。
- 所有非空成员的 `copiesPerCycle` 必须构成最简整数配比，即整组数值的最大公约数必须为 1。例如 `[2,4]` 无效，应写成 `[1,2]`。
- 空槽不参与草稿；服务端按 UI 提供的稳定槽序压缩为有序成员列表。
- 鼠标提示至少显示：成员序号、`copiesPerCycle`、样板输入输出，以及是否为待展开的闭环宏。

新编码结果不再保存独立的 `seedWaveCopies`；旧标签只有在它等于 `copiesPerCycle` 时才允许读取。一个最简循环只形成一个路由波。请求数量决定总共执行多少次循环；同时允许借用多少份种子、并行推进多少个循环波，由 `executionSeedMultiplier` 决定，不能通过给所有 `copiesPerCycle` 乘共同倍数表达。

## 4. 输出标记槽与算法结果槽

- 9 个输出槽由可用净产物候选填充，玩家可以标记其中至多 1 个为主产物、至多 8 个为副产物。
- 主产物用于确定闭环规划的请求目标；副产物只作为本闭环对外声明的附加净输出。
- 超出 9 个槽容量、没有被标记的其他产物，不进入本次主/副产物目标计算。
- 已编码闭环样板重新载入时，`payload.netOutputs().getFirst()` 对应主产物，其余已保存项按顺序作为副产物回显。
- 普通输入槽显示算法得到的 `externalInputs`。
- 种子输入槽显示算法得到的 `seeds`。
- 两类算法结果槽都只读；样板、输出标记或倍率变化后，由服务端重新计算并整体刷新。
- 如果当前声明无法形成闭环，结果槽应清空或标记为无效，不能继续显示上一份草稿的计算结果。

不新增 `primaryOutput` 字段，也不让 UI 改写 payload UUID。

## 5. 数字输入控件

- 页面提供两个正整数输入：
  - `executionSeedMultiplier`：单个任务启动时可借用多少份最简循环种子，即同时推进的循环波上限。
  - `storedTaskMultiplier`：天枢最多预存多少份任务所需种子。
- 两者最小值均为 1，与成员槽中的 `copiesPerCycle` 相互独立。
- 数字输入应允许直接键入，不要求只能通过加减按钮调整。

## 6. 服务端权威数据流

```text
插入/编码源样板
    -> 服务端运行自动发现
    -> 选中候选写入同一组 27 槽
    -> 玩家可继续增删、换序、改槽数量
    -> 玩家从最多 9 个输出候选中标记主产物和副产物
    -> 服务端按稳定槽序重建 ClosedLoopMemberPattern 草稿
    -> ClosedLoopPatternAuthoringService.createFromDraft(...)
    -> 嵌套展开后再次校验最多 27 个叶成员
    -> 精确分析并刷新只读输入槽、只读种子槽
    -> 排序、编码
```

- 推荐使用 AE2 的 ghost/config inventory 和 `FakeSlot`，由普通菜单槽同步负责客户端交互。
- 客户端只提交样板标记、输出标记和两个倍率；`SourcePatternSnapshot`、闭环执行成员判定、解码、需求计算和最终 authoring 全部在服务端完成。
- 自动候选切换必须以一次批量写入更新 27 槽，避免 change listener 在半写入状态下反复重建草稿。
- 程序填槽期间需要重入保护；填槽结束后只触发一次草稿重建和菜单同步。
- 候选成员超过 27 或展平后叶成员超过 27 时，候选不得进入可编码状态。

## 7. 与后端限制的对应关系

- UI 原始成员数：最多 27，由槽数天然限制。
- 手工 authoring：最多 27。
- 自动发现：单个候选最多 27。
- 嵌套闭环：允许作为槽成员，但展开后的普通叶成员总数仍最多 27，不能通过套娃绕过限制。
- 成员份数：展平后的整组 `copiesPerCycle` 最大公约数必须为 1；公共倍数不能保存到成员份数中。
- 输出声明：最多 1 个主产物和 8 个副产物；其余产物不参与目标计算。
- 排序仅优化所需种子量；最终是否可执行仍以完整闭环分析、模糊/过载路由校验和“每个成员至少消耗一种种子”校验为准。

UI 不应自行复制这些业务判断。数量上限应读取后端公开常量，其他失败原因由服务端状态映射为提示。

## 8. 状态与提示

至少区分以下状态：

- 没有可用候选，但仍允许玩家手工填槽。
- 成员无法解码。
- 成员数量或嵌套展开数量超限。
- 成员份数不是最简整数配比。
- 没有标记可用主产物，或输出标记无效。
- 闭环计量不成立。
- 模糊/过载种子路由不安全。
- 每个成员的输入种子类型数量不满足要求。
- 编码成功。

错误提示不能清空玩家当前槽位；切换自动候选才覆盖草稿。上传按钮的 dirty 行为不属于本次 UI 改造范围。

## 9. 建议改动点

- `Ae2ltSlotSemantics`：分别增加闭环成员、输出标记、只读普通输入和只读种子槽 semantic。
- `TianshuPatternEncodingTermMenu`：增加 27 格成员 config inventory、9 格输出标记 inventory、两组只读算法结果 inventory、两个倍率字段、批量候选填充和服务端草稿重建。
- `TianshuPatternEncodingTermScreen`：仅在闭环模式显示这些控件；具体排列由 UI 实现者决定。
- `tianshu_pattern_encoding_terminal.json`：只负责给上述控件提供最终选定的位置和样式，本设计不预设布局。

## 10. 验收用例

1. 手工放入 27 个普通成员并编码成功；第 28 个没有可放置槽。
2. 自动发现候选能原子填入同一组槽，切换候选不会残留旧成员。
3. 玩家调整某槽数量后，编码结果中的该成员 `copiesPerCycle` 与显示一致。
4. `[2,4]` 等具有公共倍数的成员份数被拒绝，改为 `[1,2]` 后可继续计算。
5. 将 `executionSeedMultiplier` 设置为 100 只扩大可借种子与并行波数，不把 `[1,2]` 改写成 `[100,200]`。
6. 玩家交换两个槽后，服务端收到的草稿顺序同步变化。
7. 一个槽引用嵌套闭环，展开为 27 个叶成员时可编码，展开为 28 个时拒绝。
8. 模糊样板与过载样板能进入槽，但不安全的动态种子路由在服务端拒绝。
9. 输出区最多接受 1 个主产物和 8 个副产物；其他产物不参与目标计算。
10. 修改样板或输出标记后，只读普通输入与种子输入由服务端重新计算，玩家不能直接修改。
11. 已编码闭环重新载入时，主产物来自 `netOutputs[0]`，副产物和两个倍率正确回显。
12. 两个倍率都可直接输入正整数，0、负数和溢出值被拒绝。
13. 普通合成、处理、过载和高级模式下，闭环页专用控件全部隐藏且不可点击。
